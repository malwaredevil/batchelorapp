---
name: AgentPhone voice webhooks need NDJSON streaming, not a single buffered JSON reply
description: Why a correct-looking `{ text }` voice webhook response can still produce silent calls, and how AgentPhone's own dedup/redelivery interacts with slow replies.
---

A voice webhook handler that returns a single buffered `{ text: ... }` JSON object (matching AgentPhone's documented "simple response" shape) can still make every turn *after* the call's opening line come back as silence once the handler's real latency (e.g. an LLM/tool-calling turn, ~1-3s) exceeds roughly one second. AgentPhone's real-time voice pipeline expects the first bytes of a response almost immediately; if nothing arrives in time it can re-deliver the same logical turn under a **new** `X-Webhook-ID`. A correct, intentional webhook replay-dedup table (inserting `X-Webhook-ID` before any side effect, per the threat model's replay-protection requirement) then does its job perfectly — but from the outside this looks like every real turn getting silently swallowed as a `"duplicate"`, because the *original* slow response is abandoned by the caller's side before it arrives.

**Why:** confirmed by pulling real webhook delivery logs from AgentPhone's `GET /v1/webhooks/deliveries` API (not just app logs) during a real live call: `agent.message` (voice) events came back with `responseBody: {"ok":true,"duplicate":true}` even though the dedup table itself was behaving correctly on each individual ID — the bug was upstream latency, not the dedup logic. Also confirmed via `GET /v1/agents` that the call's opening line comes from the agent's configured `beginMessage`, spoken directly by AgentPhone with **no webhook call at all** — so "the greeting works but follow-ups don't" is expected even when literally every webhook-driven turn is broken, since the greeting never touches the webhook.

**How to apply:** for any voice webhook handler expected to take longer than ~1s (LLM calls, tool use, DB round-trips), respond with `Content-Type: application/x-ndjson` and write an immediate `{"text": "...", "interim": true}` acknowledgement chunk (flushed right away) before doing the slow work, then write the final `{"text": "..."}` line (no `interim`) and end the response. This keeps the turn open on AgentPhone's side so it never needs to re-deliver, which also means the dedup table never sees a legitimate false-positive. Verify by replaying an exact production-shaped signed request (same HMAC secret, same body shape) against the handler directly and checking wall-clock timing of each streamed chunk — this reproduces the real behavior without needing to place an actual phone call for every iteration. Don't touch or weaken the dedup/replay-protection logic itself as a "fix" for this — a genuinely duplicated `X-Webhook-ID` must still be rejected.
