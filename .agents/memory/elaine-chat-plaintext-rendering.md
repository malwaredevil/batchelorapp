---
name: Elaine chat bubble has no markdown renderer
description: The Elaine chat UI has never had a markdown parser (no react-markdown/marked anywhere) — any ** or * syntax the model emits shows as literal characters.
---

The user reported Elaine's replies looked "like a weather widget" before and now show "a bunch of text and extra characters" in a "weird font." There was no code regression and no markdown renderer was ever removed — `lib/elaine-ui/src/ElaineChatPanel.tsx`'s `MessageText` only ever handled `[N]` citation markers and rendered everything else via `whitespace-pre-wrap`. The system prompt in `artifacts/api-server/src/elaine/index.ts` told the model to write plain text for emails but never said the visible chat reply itself had no renderer, so the model would occasionally format replies with `**bold**` / `* bullet` markdown, which rendered as literal stray asterisks (perceived as "extra characters"/"weird font").

**Why:** Squashed git history (2 commits total) meant there was no way to diff against a "working" past version — had to reason from current code + the user's screenshot alone. Ruled out a code regression by grepping for markdown libraries across the whole repo (none exist) and confirming tool result strings (e.g. weather forecast text) are already clean plain text — so the `**` was coming from the model's own reply phrasing, not tool output.

**How to apply:** Fixed both ends: (1) added an explicit FORMATTING instruction to the system prompt telling the model the chat bubble has no markdown renderer and to never use `**`, `*`, `#`, or backticks; use plain sentences or `1. / 2.` numbered lines instead. (2) added a `stripStrayMarkdown()` sanitizer in `MessageText` (and applied to the live streaming buffer) as defense-in-depth, so already-stored history and any future slip-ups still render clean. Verify by resetting the conversation (`DELETE /api/elaine/conversation`) then POSTing a fresh weather/restaurant query to `/api/elaine/chat` and checking the raw SSE `content` for stray `**`/`*` — don't rely on old stored messages in the conversation history, they'll still show the pre-fix markdown until overwritten.
