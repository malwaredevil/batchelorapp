---
name: SSE streaming with mid-stream action detection
description: Pattern for streaming a chat completion to the client while surfacing a proposed action as soon as it's ready, not just at the end. Covers both the legacy regex-directive approach and the current real tool-calling approach (elAIne/travels assistant).
---

When a chat feature needs to show a proposed action as soon as it's ready, rather than waiting for the whole reply, stream the completion with `stream: true` and detect the action incrementally as chunks arrive.

**Current approach (real tool-calling, not regex directives):** elAIne's assistant (`artifacts/api-server/src/routes/travels/assistant.ts`) migrated off embedded `[[ACTION/NAVIGATE/REMEMBER]]` bracket directives to real OpenAI-style function/tool calling. Tools are passed as a combined array (server tools + write-action tools + soft tools like `suggest_navigation`/`remember_household_fact`) to `callModelWithSubagent`. While streaming, accumulate `delta.tool_calls` by index into a `Map` (name + concatenated arguments string), and attempt to `JSON.parse` the accumulated arguments after each chunk — a fallback pass also runs once the stream ends in case the whole call arrived in one chunk. This is strictly better than regex directives: arguments are schema-validated JSON instead of hand-parsed brackets, and the model can't accidentally leak directive syntax into visible text.
- Newer `openai` SDK versions type `ChatCompletionTool` as a union (`ChatCompletionFunctionTool | ChatCompletionCustomTool`); when building a `Set` of tool names from a tools array typed as the union, cast each entry to `ChatCompletionFunctionTool` before accessing `.function.name`.
- Verified live: Gemini 2.5 Flash via OpenRouter reliably streams `delta.tool_calls` in this SDK version — plain replies, write-action proposals, and soft-tool calls (remember) all worked correctly end-to-end.
- Only surface one write-action per turn (first tool call wins via a `sent` flag) to keep the client contract (`action: null | {...}`) unchanged.

**Legacy approach (superseded, kept for reference):** the old pattern ran the *same* directive regex (e.g. matching a bracketed `[[ACTION: ...]]`) against the growing accumulated buffer after every chunk; a regex requiring the closing delimiter naturally returns no match until the directive is complete. This was replaced because bracket directives are fragile (hand-parsed, no schema validation, easy for the model to malform or leak into visible text) — real tool-calling is the preferred pattern going forward for any new streaming action-detection needs.

**How to apply (still true for both approaches):**
- OpenRouter's server tools (`openrouter:advisor`, `openrouter:subagent`) work fine combined with `stream: true` and with additional custom tools in the same array — no special handling needed, tool execution is transparent to the streamed content.
- Express: SSE requires `Content-Type: text/event-stream`, `Cache-Control: no-cache, no-transform`, `X-Accel-Buffering: no`, and `res.flushHeaders()` before writing; wrap the whole handler body in try/catch and emit an `error` SSE event + `res.end()` on failure — you can't fall back to `res.status().json()` once headers are sent.
- Client: don't force streaming endpoints through a JSON-only generated API client (e.g. orval/react-query codegen) — hand-write a small `fetch` + `ReadableStream` reader that splits on `\n\n`, parses `event:`/`data:` lines, and dispatches per event type. Keep using the generated hooks for every other (non-streaming) endpoint. The SSE event contract (`delta`/`action`/`navigate`/`done`/`error`) stayed identical across the directive→tool-calling migration, so no frontend changes were needed.
