---
name: elAIne itinerary actions
description: How add_itinerary_day / regenerate_itinerary_day were wired into the elAIne assistant action pattern, and the shared-logic extraction that made it possible.
---

The itinerary regeneration prompt/model-call logic previously lived only inline inside the `POST /trips/:id/itinerary` route handler in `ai.ts`. It was extracted into an exported `generateItineraryForTrip(tripId, style, interests, regenerateDay?)` function (throwing a typed `ItineraryActionError` instead of writing directly to `res`) so the assistant's `regenerate_itinerary_day` action tool could reuse the exact same AI logic instead of duplicating the prompt.

**Why:** `assistant.ts`'s confirm-then-execute action executors must not fork business logic that already exists in a hand-written route â€” divergence is how the two paths silently drift (e.g. different prompt wording, different day-index handling).

**How to apply:** When adding a new elAIne action that needs logic already implemented in a route handler, first extract that handler's core logic into an exported function that throws a typed error (status + message) rather than touching `res` directly, then have both the original route and the new executor call it.

Also relevant:

- `itinStyle`/`itinInterests` are per-session React state in `TripDetail.tsx`, never persisted server-side â€” the assistant executor has no way to see a user's UI style/interest picks, so it always regenerates with hardcoded defaults (`"balanced"`, `["food","history","culture"]`). The tool description explicitly tells the model this.
- The assistant tool takes a 1-based `dayNumber` (matching what's shown on screen, e.g. "Day 3") and the executor subtracts 1 before calling the shared function, which is 0-based â€” natural-language day numbers are always 1-based.
- `AssistantActionType`/`AssistantAction` in `lib/api-client-react/src/travels.ts` is a **hand-maintained** type union, not codegen'd (consistent with the already-broken orval codegen noted elsewhere) â€” it was already stale (missing `add_reminder`/`sync_reminder_to_calendar`) before this addition; when adding a new assistant action type, always update this file too or the frontend type will silently allow invalid `.type` values.

When touching `assistant.ts`, check that every frontend hook calling an `/assistant/*` endpoint has a matching server route â€” a prior session added `useDeleteHouseholdMemory()` in `Settings.tsx` calling `DELETE /assistant/memory/:id` without ever adding the server route; the gap went unnoticed until an unrelated task's completion review caught it. Frontend hooks referencing assistant endpoints are not proof the endpoint exists.

All itinerary/trip-mutating action executors intentionally scope with `and(eq(travelsTrips.id, tripId), eq(travelsTrips.userId, userId))`, even though `GET /trips` itself is household-shared (no owner filter) per `travels-shared-household-scoping.md`. This means elAIne can propose an action while viewing a trip owned by another household member, but execution 404s as "Trip not found" â€” this is existing, deliberate behavior (verified by testing `regenerate_itinerary_day` against a same-household, different-owner trip), not a bug to fix. When testing new trip-write actions via curl, always confirm the logged-in test account is the trip's actual `userId` first (check via a harmless owned-trip action like `add_itinerary_day`), or every test will 404 for the wrong reason.

`generate_itinerary` (full-trip AI itinerary generation/overwrite, added alongside `confirm_itinerary_activity`/`remove_itinerary_activity` for document-sync cleanup) reuses `generateItineraryForTrip(tripId, style, interests)` with no `regenerateDay` arg â€” the function already supported full generation, no `ai.ts` changes were needed, only a new action wrapping it. `confirm_itinerary_activity`/`remove_itinerary_activity` read-modify-write the `itinerary` jsonb column directly by 1-based day/activity index (validated in-range, 400 on out-of-range) since there's no per-activity route to reuse.

`send_email` (elAIne offers to email lists/summaries/recommendations) is the first action whose payload has NO user-supplied recipient â€” the executor always looks up the caller's own `appUsers.email` by session `userId` server-side and ignores any address the model might otherwise be tempted to accept, since this is a phishing/spam vector per the threat model's Elevation-of-Privilege concerns. The system prompt tells elAIne to only offer emailing, never call the tool until the user explicitly agrees. `systemPrompt` in `assistant.ts` is a template literal â€” backticks inside its paragraph text will silently break it into a syntax error at an unrelated line number (had to swap `` `subject` ``/`` `body` `` for plain words); never put backticks in prose added to that string.

`web_search` (elAIne live-web lookups, backed by `perplexity/sonar` via OpenRouter, see `MODELS.RESEARCH` in `ai-client.ts`) is the first tool in this file that isn't a fire-and-forget "soft" tool â€” the model needs the search result fed back before it can write its real answer. Every other tool here (navigate/remember/set*mode/the ACTION_TOOLS) is resolved entirely from the single streamed completion and never needs a second model round-trip. Implementing this required turning the previously-single streamed `client.chat.completions.create` call in the `/assistant/chat` handler into a bounded loop (`MAX_ROUNDS = 2`): stream a round, and if any `web_search` tool_calls came back, push an `{role:"assistant", tool_calls:[...]}` message plus one `{role:"tool", tool_call_id, content}` result message per call, then run one more round so the model can answer using the results. `messages` had to be explicitly typed `OpenAI.Chat.Completions.ChatCompletionMessageParam[]` (was previously untyped/inferred) since pushing `tool_calls`/`tool`-role messages doesn't fit the narrower inferred union of the original array literal. `sentActionIndices` (dedup between mid-stream and post-stream action handling) must be re-declared \_inside* the round loop, not outside it â€” OpenAI/OpenRouter restart tool-call `index` at 0 on every completion, so a Set shared across rounds can wrongly treat an action in round 2 as already-sent because round 1 happened to use the same index.

`consult_experts` (multi-model advice panel, `expert-consult.ts`) reused the same hard-tool round-trip pattern as `web_search` â€” both tool types are now collected into separate accumulator arrays each round and executed together via `Promise.all` before the next round runs, so a turn needing both a search and an opinion only costs one extra round, not two. A dedicated `"status"` SSE event type (separate from `delta`/`action`/`done`/`error`) was added end-to-end â€” server `sendEvent("status", {message})` right before executing any hard tool call, `onStatus` callback in `AssistantChatStreamCallbacks` (`lib/api-client-react/src/travels.ts`), and frontend state cleared on the next delta/done/error â€” specifically so a multi-second multi-model round doesn't look like a frozen UI (the pre-existing three-dot bubble alone wasn't enough once turns could take several seconds). For genuine multi-model diversity, the second panelist model must be a different vendor family than any model already used elsewhere in the same decision path â€” reusing the same vendor for "two opinions" isn't real independence.
