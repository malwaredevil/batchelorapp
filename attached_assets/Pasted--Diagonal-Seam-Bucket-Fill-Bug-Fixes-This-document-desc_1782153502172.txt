# Diagonal Seam & Bucket-Fill Bug Fixes

This document describes every bug that was found and fixed in the Block Designer and Whole-Quilt Designer relating to diagonal cell seams, bucket-fill behaviour, and fabric rendering.

---

## Bug 1 — Bucket-fill on a 1×1 block filled the whole cell instead of just the clicked half

### Symptom
When a block was exactly 1×1 in size (a single cell) and a horizontal or vertical seam was drawn through it, using the bucket-fill tool would fill the **entire** cell with one colour instead of just the half on the clicked side of the seam.

### Root cause
The `seamFill` function's flood-fill (BFS) works in two layers:

1. **Cross-cell boundaries** — tracked in `hEdges`/`vEdges` using *even* position numbers (e.g. position 2 = the boundary between row 0 and row 1).
2. **Within-cell midline seams** — H/V seams drawn along a cell's internal midline have *odd* position numbers (e.g. position 1 = the midline of the only cell in a 1×1 grid).

Odd-position midline seams were only stored in `hEdges`/`vEdges`; they were never consulted during the BFS because the BFS only read `cellSeps()` (which returns the cell's own shape separators from the stored cell string, not from the `seams` array). Since a freshly divided cell is encoded as `"seam-midline-h"` or `"seam-midline-v"` only *after* being filled — not before — the midline seam in the `seams` array was completely ignored by the BFS, so the fill flooded the whole cell.

A 1×1 grid has no internal boundaries other than its midline, so the bug was 100% reproducible on 1×1 blocks. Multi-cell grids had enough cross-cell boundaries to mask it in most cases.

### Fix — `artifacts/pottery/src/pages/blocks/designer.tsx`
Added a new `seamSeps: Map<number, Set<Sep>>` data structure built from full-span odd-position midline seams in the `seams` array:

- A **horizontal** midline seam at odd position `pos` → adds `"hmid"` to the per-cell sep set for every cell in row `r = (pos - 1) / 2`.
- A **vertical** midline seam at odd position `pos` → adds `"vmid"` to the per-cell sep set for every cell in column `c = (pos - 1) / 2`.
- Clipped seams (where `cs ≈ 0` and `ce ≈ 1` is not true) are skipped — a partial seam cannot enclose a region.

The internal `getSeps(cellIndex)` helper was updated to merge `seamSeps.get(cellIndex)` into the set of separators used by the BFS. This means the BFS now stops at midline seams even in a 1×1 cell.

## Bug 2 — Fabric fills appeared as light grey on single diagonal cells (and "retroactively fixed" when a larger area was filled)

### Symptom
In any grid size, drawing a diagonal seam line in **one** cell and then using the bucket-fill with a fabric selected would show the filled triangle as **light grey** instead of the fabric texture. However, if a large multi-cell diagonal fill was done that happened to include some **solid** fabric cells, the previously-grey single-cell triangle would suddenly display the fabric correctly.

### Root cause
SVG fabric fills work via `<pattern>` elements declared in `<defs>`. CellShape resolves a fabric token like `"fab:42"` to `url(#fab-42)`, which references one of these declared patterns.

The `<defs>` were built by scanning the `cells` array with:

```js
if (c.startsWith("fab:")) { /* add pattern for this ID */ }
```

This check only caught **solid** fabric cells (encoded as just `"fab:42"`). A triangle encoded as `"nesw:fab:42:"` does **not** start with `"fab:"` — it starts with `"nesw:"`. So when a single diagonal cell was the only thing using that fabric, no `<pattern id="fab-42">` was ever declared in the SVG, and `url(#fab-42)` was unresolved → the triangle rendered as transparent/grey.

The "retroactive fix" happened because a large fill turns interior cells into solid `"fab:42"` strings, which *do* start with `"fab:"` → the pattern gets declared → every `url(#fab-42)` in the SVG, including the existing single-cell triangle, now resolves → the triangle renders correctly.

The same bug existed in both the **editable grid** (`BlockGrid`) and the **tiled preview** (`BlockPreview`).

The SVG export (`svg-export.ts`) was **not** affected — it already used a correct regex-based `extractFabricIds()` function.

### Fix — `artifacts/pottery/src/pages/blocks/designer.tsx`
Replaced the `startsWith("fab:")` check in both scanning loops with a regex that finds all `fab:{N}` tokens anywhere in a cell string, regardless of position:

```js
const FAB_RE = /fab:(\d+)/g;
for (const c of cells) {
  let m: RegExpExecArray | null;
  FAB_RE.lastIndex = 0;
  while ((m = FAB_RE.exec(c)) !== null) {
    const n = parseInt(m[1], 10);
    if (!isNaN(n) && fabricUrlMap[n]) ids.add(n);
  }
}
```

This correctly handles all cell encodings that can contain a fabric colour:

| Cell encoding | Example | Caught before? | Caught after? |
|---|---|---|---|
| Solid fabric | `fab:42` | ✅ | ✅ |
| Triangle A | `nesw:fab:42:` | ❌ | ✅ |
| Triangle B | `nwse:#FF0000:fab:42` | ❌ | ✅ |
| Quad face | `quad:fab:42:fab:5:#FFF:#000` | ❌ | ✅ |
| H-split half | `hsplit:fab:42:#FFF` | ❌ | ✅ |
| V-split half | `vsplit:#FFF:fab:42` | ❌ | ✅ |

## Bug 3 — Fabric hover popup appeared below the canvas (iframe overflow)

### Symptom
In both the Block Designer and Whole-Quilt Designer, hovering over a fabric in the fabric picker sidebar caused a magnified 160×160 preview popup to appear **below the canvas** area. Because the app runs inside an iframe (in Replit's preview pane), the popup's `position: absolute; right: 100%` positioning escaped the normal scroll bounds and appeared outside the expected region.

### Fix — `artifacts/pottery/src/components/FabricPicker.tsx`
Removed the hover preview popup entirely:

- Removed the `hoveredId` state and `hoverTimer` ref.
- Removed the `onEnter` / `onLeave` handler functions.
- Removed `onMouseEnter` / `onMouseLeave` from the fabric list item `<div>`.
- Removed the `isHovered` constant.
- Removed the popup JSX (`{isHovered && fabric.imageUrl && <div ...>...</div>}`).

The fabric thumbnail (36×36 px) remains visible inline in the list for identification. The fabric name remains visible as text. Hovering no longer triggers any additional UI.
