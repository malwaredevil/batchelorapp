# Threat Model

## Project Overview

Batchelor App is a public internet-facing pnpm monorepo that serves a shared Express 5 API plus multiple React/Vite web artifacts under one domain. The deployed API currently mounts pottery, quilting, travels, hub, and shared auth routes. Users authenticate with shared email/password or Google OAuth, then access data stored in a shared Supabase Postgres database and private Supabase Storage buckets. Pottery, quilting, and travels data are all **fully household-shared**: every authenticated household member can view, create, edit, and delete any pottery/quilting record, shopping-list item, trip, wishlist entry, photo, reminder, trip document, custom document type, or calendar-derived trip suggestion, regardless of which user originally created it. `user_id` columns on these tables are retained only to record which household member created a record (insert attribution / audit trail) — they are never used to filter, scope, or gate reads, writes, or deletes. This intentionally supersedes any earlier per-account isolation model. The only travel-related surfaces that remain single-owner are the ones tied directly to a personal OAuth grant (Google Calendar connections/tokens, Gmail connections/tokens/scan-decisions), plus purely personal UI state (dashboard card layout/collapse prefs, account settings) — see the Household-sharing boundary below for the exact line. One user-dismissed exception now exists for Google Calendar: if a household owner deliberately designates one connected calendar row as the shared Travel calendar, the family accepts that calendar's events and write side effects becoming household-visible/household-mutable through the app. Future scans should not re-propose that exact shared-Travel-calendar proxy behavior unless the designation gate broadens or a new path reaches non-designated personal calendars. Travels also includes a bearer-token share-link route (`GET /api/travels/trips/:id/share?token=...`) that serves a limited trip view without a session; this must be modeled explicitly as a public boundary exception rather than assumed to be covered by the normal authenticated-only travels routes. `artifacts/mockup-sandbox/` is development-only unless future deployment wiring exposes it.

The travels app also includes an opt-in Gmail scanning feature: each household member may separately connect their own Gmail account (`gmail.readonly` + `gmail.labels` scopes, per-user OAuth tokens distinct from the Calendar OAuth grant) so the app can auto-detect flight/train/hotel confirmation emails, AI-extract structured trip data, and suggest linking them as trip documents. A companion manual mode lets a connected user browse/search their own inbox directly and choose any email to link. All Gmail access is strictly scoped to the connecting user's own tokens and own inbox — there is no cross-user Gmail access, even though trips themselves are household-shared. The `gmail.labels` scope grants a narrow write capability: the server may create/apply two server-managed labels ("Travel", "Batchelor App") on the user's own messages, but only when the user accepts an AI-generated scan suggestion as travel-related — manually linking or ignoring a message from inbox search never applies labels, since that path reflects the user finding an email themselves rather than confirming an AI classification. It never modifies message content, never trashes/deletes/archives messages, and label application is best-effort (failures are logged and swallowed, never surfaced as a user-facing error or allowed to block the underlying link/ingest operation).

## Assets

- **User accounts and sessions** — email addresses, bcrypt password hashes, session cookies, Google-authenticated identities, and password-reset tokens. Compromise enables full account takeover across both apps.
- **Collection data** — pottery items, quilting fabrics/patterns/quilts/layouts/shopping data, notes, categories, and derived AI metadata. This is authenticated-only content (never exposed to unauthenticated requests) but is intentionally fully shared and mutable across every household account — there is no per-user ownership boundary within pottery/quilting. `user_id` values are attribution metadata only.
- **Private images** — photos stored in private Supabase buckets for both apps. Leakage to unauthenticated parties would expose collection images and derivative metadata; access among authenticated household members is intentionally unrestricted, consistent with the shared collection-data model.
- **Travel plans and documents** — trips, reminders, uploaded travel documents/photos, calendar connections, and itinerary data. Some travel data is intentionally shared within a household account set, but uploaded documents, storage objects, OAuth tokens, and settings still require explicit server-side scoping.
- **Travel share tokens** — bearer tokens stored on `travels_trips.share_token` that unlock the public trip-share endpoint without a session. Anyone who obtains a token can read the shared subset of that itinerary until the token is revoked, so token leakage is equivalent to disclosure of that trip snapshot.
- **Application secrets** — Supabase service-role key, database credentials, session secret, OAuth client secrets, Resend credentials, and AI provider keys. Server-side secret compromise would expose all tenant data and storage.
- **AI-backed compute and outbound integrations** — OpenAI/OpenRouter/Jina/Voyage calls, password-reset email delivery, Google OAuth exchanges, RSS fetching, and any server-side URL fetching. Abuse can create quota burn, SSRF-style internal access, or confidentiality leaks through third-party requests.
- **Gmail OAuth tokens and inbox content** — per-user Gmail refresh/access tokens (`gmail.readonly` + `gmail.labels` scopes, separate grant from Calendar), plus any email subject/body/attachment content fetched during auto-scan or manual browsing. Compromise would expose a user's personal inbox contents, allow continued read access until the grant is revoked, and allow creation/application of the two server-managed labels ("Travel", "Batchelor App") on the user's own messages; leakage across users would violate the single-owner Gmail access guarantee even though trips are household-shared.
- **AgentPhone webhook secret and SMS/voice channel** — a shared HMAC secret (`AGENTPHONE_WEBHOOK_SECRET`) authenticates inbound `agent.message` events for the household's one provisioned SMS/voice number. Compromise of the secret would let an attacker forge inbound messages/calls and reach the restricted Elaine action executor as an arbitrary phone-matched household member. The webhook also grants Elaine a narrow, non-destructive write capability over trips/reminders/packing lists purely from an unauthenticated phone channel (authenticated only by possession of a recognized phone number plus the shared webhook secret) — a materially weaker identity proof than the session-cookie login used everywhere else in the app.

## Trust Boundaries

- **Browser to API** — all client input, uploaded images, URLs, IDs, and search/filter parameters are untrusted and cross into the Express API.
- **API to session store / database** — the API has direct write access to shared Supabase-backed state. Any injection or authorization failure can expose or corrupt both apps' data.
- **API to private object storage** — the server uses a Supabase service-role client to read/write private buckets. Access control is fully enforced in application code.
- **API to third-party services** — the server exchanges secrets with Google OAuth, Resend, OpenAI/OpenRouter/Jina/Voyage, performs server-side URL fetches for pattern import and hub RSS, and talks to Google Calendar APIs. These integrations are high-value SSRF and data-exfiltration boundaries.
- **Public to authenticated surfaces** — health and auth entry points are public; most pottery, quilting, travels, and hub routes are authenticated, but the travels share-link read endpoint is intentionally public when presented with a valid bearer token. There is no public self-signup route in the current API, so abuse findings should distinguish public unauthenticated attack paths from account-holder abuse paths.
- **App-to-app shared boundary** — Pottery and Quilting share one user table, one session model, and one Supabase project with namespaced tables/buckets. Namespacing must stay intentional (pottery routes touch only `pottery_*` tables/buckets, quilting routes touch only `quilting_*`), even though within each app all authenticated household accounts have full shared access.
- **Household-sharing boundary** — pottery, quilting, and travels all intentionally expose shared household data across every authenticated user (full read/write/delete, not just read). Within travels this now explicitly includes trips, wishlist entries, photos, reminders, trip documents, custom document types, and calendar-derived trip suggestions (`travels_calendar_trip_suggestions`) — a suggestion sourced from one member's personal calendar scan is visible/dismissible/acceptable by any household member, since the suggestion itself is trip data, not the calendar connection. The household has also explicitly accepted one narrower Google Calendar exception: once a household owner designates a connected calendar row as the shared Travel calendar, that specific calendar is treated as a shared family resource inside the app even though the underlying OAuth grant belongs to one member. This accepted-risk exception does not widen the rest of the boundary: routes must still require a valid authenticated session (no unauthenticated access), storage objects must still be fetched only through server-side authenticated handlers, non-designated OAuth tokens/connections (Google Calendar, Gmail) plus purely personal UI state (dashboard card layout/collapse prefs, account settings) remain single-owner, and "household-shared" still means shared _among authenticated household accounts_, never shared with unauthenticated requests.
- **Public share-token boundary** — `GET /api/travels/trips/:id/share?token=...` is a separate bearer-secret boundary from the household session model. The token is the only gate on that route, so it must stay high-entropy, revocable, and deliberately scoped to the exact fields intended for outside sharing; it must never be confused with the normal authenticated-only household-sharing model.
- **Dev-only to production boundary** — `artifacts/mockup-sandbox/`, build outputs, and local tooling are out of production scope unless explicitly wired into the deployed artifact.
- **API to Gmail** — the server holds per-user Gmail OAuth tokens and calls the Gmail API (search/fetch messages, fetch attachments, list/create/apply labels) on the connecting user's behalf. This boundary must never let one user's session or trip fetch another user's Gmail data, must validate OAuth state/tokens the same way as the existing Calendar grant, must not let inbox content (subjects/snippets/attachments) leak into another user's suggestions, logs, or AI prompts beyond what's needed for extraction, and the label-write capability must remain scoped to only the two server-managed labels on the connecting user's own messages — never message deletion/trash/content modification, and never triggered by anything other than the user's own explicit confirmation of an AI suggestion.
- **AgentPhone to API** (`/api/agentphone/webhook`) — this is the only mounted route in the app that authenticates a request by something other than a session cookie or OAuth token: it trusts an HMAC-SHA256 signature (`X-Webhook-Signature: sha256=<hex>` over `{X-Webhook-Timestamp}.{raw body}`, keyed by `AGENTPHONE_WEBHOOK_SECRET`) plus a timestamp freshness check (rejects requests older than 5 minutes) and per-delivery dedup (`X-Webhook-ID`, recorded in `agentphone_webhook_deliveries` before any side effect, so a redelivered webhook is a no-op). Signature verification must run on the exact raw request bytes (captured via a path-scoped `express.json({verify})` mounted before the app-wide body parser), not a re-serialized body, or a byte-for-byte-different-but-semantically-equal payload could bypass the HMAC check. The webhook maps inbound SMS/voice to a household member purely by matching the caller's phone number against `app_users.phone_number` — there is no further per-message authentication, so this phone-number match is the full identity boundary for this channel; an unrecognized number gets no reply (SMS) or an immediate decline+hangup (voice) rather than any household data. Once matched, the webhook hands the message to a **restricted** Elaine turn (`runAgentphoneTurn`, `auto_run` mode) whose tool set is a hard allowlist of non-destructive travel actions (add/edit reminder, sync reminder to calendar, add/remove packing item, update trip status) — it must never gain access to the full in-app Elaine tool set (email sending, web search, document/photo actions, destructive deletes) without a deliberate, reviewed allowlist change. STOP/HELP/START (10DLC) keyword handling must run before any Elaine turn and must work for every inbound message regardless of phone-number recognition or current opt-out state, since it is a carrier compliance requirement, not an app feature.

## Scan Anchors

- **Production entry points**: `artifacts/api-server/src/index.ts`, `artifacts/api-server/src/app.ts`, `artifacts/api-server/src/routes/**`
- **Highest-risk code areas**: shared auth/session/password-reset flow, authentication enforcement on pottery/quilting/travels routes (all data is household-shared, not per-owner — the risk is unauthenticated access, not cross-account access), the public travels share-token path, private image/storage access, AI-assisted import/analysis routes, hub RSS fetching, any raw SQL or external fetch logic, Gmail OAuth/token storage + message/attachment fetching (which remain single-owner even though pottery/quilting/travels data is shared), Google Calendar paths outside the explicitly accepted shared-Travel-calendar proxy behavior, and the AgentPhone webhook signature/replay verification plus its restricted-action allowlist (`artifacts/api-server/src/routes/agentphone.ts`, `runAgentphoneTurn` in `artifacts/api-server/src/elaine/index.ts`)
- **Public surfaces**: `/api/healthz`, `/api/auth/login`, `/api/auth/forgot-password`, `/api/auth/reset-password`, `/api/auth/google`, `/api/auth/google/callback`, `/api/auth/providers`, `/api/travels/trips/:id/share?token=...`, `/api/agentphone/webhook` (unauthenticated by session, but signature-gated — see the AgentPhone trust boundary above)
- **Authenticated surfaces**: `/api/pottery/**`, `/api/quilting/**`, almost all `/api/travels/**` (including `/api/travels/gmail/**` and `/api/travels/settings`) except the public share-link read route, `/api/hub/**`, `/api/auth/me`, `/api/auth/change-password`
- **Usually dev-only**: `artifacts/mockup-sandbox/**`, generated `dist/**`, repo tooling/scripts unless they are reachable through deployed API behavior

## Threat Categories

### Spoofing

The application relies on session cookies and shared account records to identify users across both apps. All protected endpoints must require a valid server-side session, session identifiers must be regenerated on login, OAuth callbacks must validate anti-CSRF state and Google ID tokens, and password-reset flows must not allow attackers to impersonate users through token reuse or enumeration. The Gmail OAuth flow follows the same pattern as Calendar: the `/api/travels/gmail/connect` → `/callback` round trip must validate a signed, single-use state cookie before exchanging the authorization code, so an attacker cannot force-link their own Gmail grant onto a victim's session (CSRF) or replay a stale callback. The AgentPhone webhook has no session at all, so spoofing there means forging a request AgentPhone never sent: every request must pass HMAC signature verification over the raw body plus a bounded timestamp window (rejects requests older than 5 minutes) before any household data is touched, and a valid signature with a reused `X-Webhook-ID` must be a no-op rather than reprocessed (replay). A caller who passes signature verification is still only ever treated as the household member matching the inbound phone number — there is no additional per-message identity claim to spoof beyond that number match.

### Tampering

Users can submit structured JSON, uploaded images, route parameters, and externally hosted URLs that influence stored records and AI-generated metadata. The server must validate all request bodies with strict schemas and require a valid authenticated session on every read/write/delete path for pottery, quilting, and travels data — by design, any authenticated household member may create, edit, or delete any record in these apps (there is no per-user ownership check to enforce), so the control that matters is authentication, not authorization-by-owner. Storage paths or AI-derived fields must not be usable to tamper with unrelated data outside the intended tables/buckets. Gmail message IDs, dismiss/link/ignore actions, and timezone values submitted by the client must be validated and scoped to the requesting user's own Gmail connection/decisions before any write — this remains single-owner even though the resulting trip data is shared. AgentPhone SMS/voice input (free-text message body or voice transcript) is untrusted content reaching the restricted Elaine turn; the tampering control here is the tool allowlist itself (`AGENTPHONE_ACTION_TYPES`) — the model may only invoke a fixed, non-destructive set of travel actions (add/edit reminder, sync-to-calendar, add/remove packing item, update trip status), never delete/send-email/document/photo actions, regardless of what the inbound text asks for.

### Information Disclosure

This project stores collection data, travel documents, and object-storage images behind authenticated API routes, while also holding high-value server secrets for Supabase and AI providers. Pottery and quilting records are intentionally visible to every authenticated household member — there is no "other user's private record" concept within these apps — so the disclosure risk to guard against is exposure to _unauthenticated_ requests, not cross-account exposure within the household. API responses, image/document download handlers, logs, and third-party integration calls must never expose secrets, raw password-reset artifacts, private bucket contents to unauthenticated callers, or OAuth/calendar/Gmail tokens. The public travels share-link route is a deliberate exception only to the extent of the fields it is meant to expose; its bearer token must therefore be treated as sensitive trip data and must not be logged, embedded in referers unnecessarily, or broadened to additional fields without review. Public routes must avoid account enumeration and verbose error leakage. Gmail access tokens/refresh tokens must never be returned in API responses, and every Gmail read (status, inbox search, message fetch, attachment fetch, suggestions) must use only the requesting user's own stored token — never another household member's — even though the resulting trip suggestion may itself be linked to a shared trip. The webhook route (`/api/agentphone/webhook`) must never echo `AGENTPHONE_WEBHOOK_SECRET`, another household member's phone number, or full Gmail/OAuth token values back into an SMS reply or voice `{text}` response, and error responses on signature failure must stay generic (no hints about why verification failed) to avoid helping an attacker iterate toward a forged signature.

### Denial of Service

The API accepts login attempts, image uploads, image-derived AI analysis, bulk re-analysis, RSS fetching, and server-side URL imports, all of which can be abused for cost amplification or resource exhaustion. Public and authenticated expensive endpoints must be rate-limited at the deployment boundary, upload sizes and image expansion must be bounded, and outbound network requests must use strict timeouts, redirect controls, destination validation, and content limits. Gmail auto-scan and AI extraction add another cost-amplification surface (Gmail API calls plus per-message AI extraction calls); scans must be bounded (page size/time window), avoid unbounded re-scanning of already-decided messages, and manual inbox search/fetch must not allow arbitrarily large or repeated result sets to drive excessive Gmail API or AI usage. The Gmail decision ledger is part of that control surface: ignore/dismiss/reconsider flows must not let a user reset a previously linked message into an unhandled state without going through the explicit unlink path, or the same source email can be re-imported repeatedly. The bulk-link endpoint (`POST /gmail/messages/bulk-link`) amplifies this further by fanning one request out into N per-message Gmail fetches + AI extraction + document creations, and (when a message has multiple attachments) N-times-attachment-count document writes; it must stay capped by a fixed per-request max message count (currently 25) and must not be reachable with an unbounded or user-controlled batch size. The AgentPhone webhook is a per-message AI-completion cost-amplification surface reachable without a session: the request body is capped at 64kb, the restricted Elaine turn is capped at 3 tool-call rounds, and conversation history persisted per phone number is capped at 20 messages — even a signature-valid but abusive sender (e.g. a compromised secret, or AgentPhone itself misbehaving) cannot drive unbounded per-request cost or unbounded stored state growth.

### Elevation of Privilege

The highest-likelihood privilege failure here is an authentication bypass that lets an unauthenticated request reach pottery/quilting/travels data — since these apps are fully household-shared by design, there is no per-owner boundary to violate _within_ an authenticated household account, so `requireAuth` (or equivalent session checks) on every route is the control that matters, not per-row ownership filters. The main exception is the public travels share-token route, whose privilege boundary is possession of the bearer token rather than a session; that token must never be mintable or reusable in ways that silently broaden access beyond the intended public trip snapshot. The remaining elevation risks are SSRF or injection flaws that let an authenticated user pivot into server-side capabilities beyond the shared-data model (e.g. reaching another user's single-owner Gmail/Calendar tokens, internal network destinations, or raw SQL injection). Raw SQL must remain parameterized, and any server-side fetch or redirect logic must reject internal destinations and untrusted authority changes. Gmail-specific: connection lookups, decisions (dismiss/link/ignore/reconsider), and disconnect must all filter by the session user's own `userId`, never by trip or household membership, since Gmail access itself must remain single-owner even where the resulting trip data is shared. The reconsider endpoint (`POST /gmail/messages/:messageId/reconsider`) reverses a prior ignore/dismiss decision and must only ever touch the requesting user's own decision row (scoped by `userId` + `gmailMessageId`), must reject reconsidering an already-linked message (409), and must 404 when no matching decision exists — it must never be usable to reset a previously linked message back into an unhandled state without the unlink flow tearing down the existing import. Label management inherits the same constraint: label IDs are cached per-connection row and every `labels.list`/`create`/`modify` call must use the `userId` to retrieve the correct token and cached IDs, never assuming one user's label configuration applies to another.

AgentPhone elevation risks: The HMAC signature + phone number match is a high-privilege boundary that must never be weakened. A successful phone-number match grants access to a **restricted** Elaine turn — the elevation risk is "tool-kit leakage," where a phone user tricks Elaine into using a tool from the full app-wide tool set (e.g., `send_email`, `delete_pottery_item`, `search_web`). The control is the hard allowlist in the executor (`AGENTPHONE_ACTION_TYPES` in `artifacts/api-server/src/elaine/index.ts`): any tool call from an AgentPhone session that is not in the allowlist must be rejected before execution, regardless of the model's confidence or the user's intent. Finally, the SMS/voice channel must never be usable to initiate a password reset or OAuth connection, as it lacks the necessary browser-based proof-of-control.
