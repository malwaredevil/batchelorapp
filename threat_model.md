# Threat Model

## Project Overview

Batchelor App is a public internet-facing pnpm monorepo that serves a shared Express 5 API plus multiple React/Vite web artifacts under one domain. The deployed API currently mounts pottery, quilting, travels, hub, and shared auth routes. Users authenticate with shared email/password or Google OAuth, then access data stored in a shared Supabase Postgres database and private Supabase Storage buckets. Pottery, quilting, and travels data are all **fully household-shared**: every authenticated household member can view, create, edit, and delete any pottery/quilting record, shopping-list item, trip, wishlist entry, photo, reminder, trip document, custom document type, or calendar-derived trip suggestion, regardless of which user originally created it. `user_id` columns on these tables are retained only to record which household member created a record (insert attribution / audit trail) — they are never used to filter, scope, or gate reads, writes, or deletes. This intentionally supersedes any earlier per-account isolation model. The only travel-related surfaces that remain single-owner are the ones tied directly to a personal OAuth grant (Google Calendar connections/tokens, Gmail connections/tokens/scan-decisions), plus purely personal UI state (dashboard card layout/collapse prefs, account settings) — see the Household-sharing boundary below for the exact line. `artifacts/mockup-sandbox/` is development-only unless future deployment wiring exposes it.

The travels app also includes an opt-in Gmail scanning feature: each household member may separately connect their own Gmail account (`gmail.readonly` + `gmail.labels` scopes, per-user OAuth tokens distinct from the Calendar OAuth grant) so the app can auto-detect flight/train/hotel confirmation emails, AI-extract structured trip data, and suggest linking them as trip documents. A companion manual mode lets a connected user browse/search their own inbox directly and choose any email to link. All Gmail access is strictly scoped to the connecting user's own tokens and own inbox — there is no cross-user Gmail access, even though trips themselves are household-shared. The `gmail.labels` scope grants a narrow write capability: the server may create/apply two server-managed labels ("Travel", "Batchelor App") on the user's own messages, but only when the user accepts an AI-generated scan suggestion as travel-related — manually linking or ignoring a message from inbox search never applies labels, since that path reflects the user finding an email themselves rather than confirming an AI classification. It never modifies message content, never trashes/deletes/archives messages, and label application is best-effort (failures are logged and swallowed, never surfaced as a user-facing error or allowed to block the underlying link/ingest operation).

## Assets

- **User accounts and sessions** — email addresses, bcrypt password hashes, session cookies, Google-authenticated identities, and password-reset tokens. Compromise enables full account takeover across both apps.
- **Collection data** — pottery items, quilting fabrics/patterns/quilts/layouts/shopping data, notes, categories, and derived AI metadata. This is authenticated-only content (never exposed to unauthenticated requests) but is intentionally fully shared and mutable across every household account — there is no per-user ownership boundary within pottery/quilting. `user_id` values are attribution metadata only.
- **Private images** — photos stored in private Supabase buckets for both apps. Leakage to unauthenticated parties would expose collection images and derivative metadata; access among authenticated household members is intentionally unrestricted, consistent with the shared collection-data model.
- **Travel plans and documents** — trips, reminders, uploaded travel documents/photos, calendar connections, and itinerary data. Some travel data is intentionally shared within a household account set, but uploaded documents, storage objects, OAuth tokens, and settings still require explicit server-side scoping.
- **Application secrets** — Supabase service-role key, database credentials, session secret, OAuth client secrets, Resend credentials, and AI provider keys. Server-side secret compromise would expose all tenant data and storage.
- **AI-backed compute and outbound integrations** — OpenAI/OpenRouter/Jina/Voyage calls, password-reset email delivery, Google OAuth exchanges, RSS fetching, and any server-side URL fetching. Abuse can create quota burn, SSRF-style internal access, or confidentiality leaks through third-party requests.
- **Gmail OAuth tokens and inbox content** — per-user Gmail refresh/access tokens (`gmail.readonly` + `gmail.labels` scopes, separate grant from Calendar), plus any email subject/body/attachment content fetched during auto-scan or manual browsing. Compromise would expose a user's personal inbox contents, allow continued read access until the grant is revoked, and allow creation/application of the two server-managed labels ("Travel", "Batchelor App") on the user's own messages; leakage across users would violate the single-owner Gmail access guarantee even though trips are household-shared.

## Trust Boundaries

- **Browser to API** — all client input, uploaded images, URLs, IDs, and search/filter parameters are untrusted and cross into the Express API.
- **API to session store / database** — the API has direct write access to shared Supabase-backed state. Any injection or authorization failure can expose or corrupt both apps' data.
- **API to private object storage** — the server uses a Supabase service-role client to read/write private buckets. Access control is fully enforced in application code.
- **API to third-party services** — the server exchanges secrets with Google OAuth, Resend, OpenAI/OpenRouter/Jina/Voyage, performs server-side URL fetches for pattern import and hub RSS, and talks to Google Calendar APIs. These integrations are high-value SSRF and data-exfiltration boundaries.
- **Public to authenticated surfaces** — health and auth entry points are public; pottery, quilting, travels, and hub routes are authenticated. There is no public self-signup route in the current API, so abuse findings should distinguish public unauthenticated attack paths from account-holder abuse paths.
- **App-to-app shared boundary** — Pottery and Quilting share one user table, one session model, and one Supabase project with namespaced tables/buckets. Namespacing must stay intentional (pottery routes touch only `pottery_*` tables/buckets, quilting routes touch only `quilting_*`), even though within each app all authenticated household accounts have full shared access.
- **Household-sharing boundary** — pottery, quilting, and travels all intentionally expose shared household data across every authenticated user (full read/write/delete, not just read). Within travels this now explicitly includes trips, wishlist entries, photos, reminders, trip documents, custom document types, and calendar-derived trip suggestions (`travels_calendar_trip_suggestions`) — a suggestion sourced from one member's personal calendar scan is visible/dismissible/acceptable by any household member, since the suggestion itself is trip data, not the calendar connection. This shared-access model does not relax protections at the boundary itself: routes must still require a valid authenticated session (no unauthenticated access), storage objects must still be fetched only through server-side authenticated handlers, and OAuth tokens/connections (Google Calendar, Gmail) plus purely personal UI state (dashboard card layout/collapse prefs, account settings) remain single-owner and are never widened to household-shared even though the trip/collection data they relate to is shared. "Household-shared" means shared _among authenticated household accounts_, never shared with unauthenticated requests.
- **Dev-only to production boundary** — `artifacts/mockup-sandbox/`, build outputs, and local tooling are out of production scope unless explicitly wired into the deployed artifact.
- **API to Gmail** — the server holds per-user Gmail OAuth tokens and calls the Gmail API (search/fetch messages, fetch attachments, list/create/apply labels) on the connecting user's behalf. This boundary must never let one user's session or trip fetch another user's Gmail data, must validate OAuth state/tokens the same way as the existing Calendar grant, must not let inbox content (subjects/snippets/attachments) leak into another user's suggestions, logs, or AI prompts beyond what's needed for extraction, and the label-write capability must remain scoped to only the two server-managed labels on the connecting user's own messages — never message deletion/trash/content modification, and never triggered by anything other than the user's own explicit confirmation of an AI suggestion.

## Scan Anchors

- **Production entry points**: `artifacts/api-server/src/index.ts`, `artifacts/api-server/src/app.ts`, `artifacts/api-server/src/routes/**`
- **Highest-risk code areas**: shared auth/session/password-reset flow, authentication enforcement on pottery/quilting/travels routes (all data is household-shared, not per-owner — the risk is unauthenticated access, not cross-account access), private image/storage access, AI-assisted import/analysis routes, hub RSS fetching, any raw SQL or external fetch logic, and Gmail OAuth/token storage + message/attachment fetching (which remain single-owner even though pottery/quilting/travels data is shared)
- **Public surfaces**: `/api/healthz`, `/api/auth/login`, `/api/auth/forgot-password`, `/api/auth/reset-password`, `/api/auth/google`, `/api/auth/google/callback`, `/api/auth/providers`
- **Authenticated surfaces**: `/api/pottery/**`, `/api/quilting/**`, `/api/travels/**` (including `/api/travels/gmail/**` and `/api/travels/settings`), `/api/hub/**`, `/api/auth/me`, `/api/auth/change-password`
- **Usually dev-only**: `artifacts/mockup-sandbox/**`, generated `dist/**`, repo tooling/scripts unless they are reachable through deployed API behavior

## Threat Categories

### Spoofing

The application relies on session cookies and shared account records to identify users across both apps. All protected endpoints must require a valid server-side session, session identifiers must be regenerated on login, OAuth callbacks must validate anti-CSRF state and Google ID tokens, and password-reset flows must not allow attackers to impersonate users through token reuse or enumeration. The Gmail OAuth flow follows the same pattern as Calendar: the `/api/travels/gmail/connect` → `/callback` round trip must validate a signed, single-use state cookie before exchanging the authorization code, so an attacker cannot force-link their own Gmail grant onto a victim's session (CSRF) or replay a stale callback.

### Tampering

Users can submit structured JSON, uploaded images, route parameters, and externally hosted URLs that influence stored records and AI-generated metadata. The server must validate all request bodies with strict schemas and require a valid authenticated session on every read/write/delete path for pottery, quilting, and travels data — by design, any authenticated household member may create, edit, or delete any record in these apps (there is no per-user ownership check to enforce), so the control that matters is authentication, not authorization-by-owner. Storage paths or AI-derived fields must not be usable to tamper with unrelated data outside the intended tables/buckets. Gmail message IDs, dismiss/link/ignore actions, and timezone values submitted by the client must be validated and scoped to the requesting user's own Gmail connection/decisions before any write — this remains single-owner even though the resulting trip data is shared.

### Information Disclosure

This project stores collection data, travel documents, and object-storage images behind authenticated API routes, while also holding high-value server secrets for Supabase and AI providers. Pottery and quilting records are intentionally visible to every authenticated household member — there is no "other user's private record" concept within these apps — so the disclosure risk to guard against is exposure to _unauthenticated_ requests, not cross-account exposure within the household. API responses, image/document download handlers, logs, and third-party integration calls must never expose secrets, raw password-reset artifacts, private bucket contents to unauthenticated callers, or OAuth/calendar/Gmail tokens. Public routes must avoid account enumeration and verbose error leakage. Gmail access tokens/refresh tokens must never be returned in API responses, and every Gmail read (status, inbox search, message fetch, attachment fetch, suggestions) must use only the requesting user's own stored token — never another household member's — even though the resulting trip suggestion may itself be linked to a shared trip.

### Denial of Service

The API accepts login attempts, image uploads, image-derived AI analysis, bulk re-analysis, RSS fetching, and server-side URL imports, all of which can be abused for cost amplification or resource exhaustion. Public and authenticated expensive endpoints must be rate-limited at the deployment boundary, upload sizes and image expansion must be bounded, and outbound network requests must use strict timeouts, redirect controls, destination validation, and content limits. Gmail auto-scan and AI extraction add another cost-amplification surface (Gmail API calls plus per-message AI extraction calls); scans must be bounded (page size/time window), avoid unbounded re-scanning of already-decided messages, and manual inbox search/fetch must not allow arbitrarily large or repeated result sets to drive excessive Gmail API or AI usage. The bulk-link endpoint (`POST /gmail/messages/bulk-link`) amplifies this further by fanning one request out into N per-message Gmail fetches + AI extraction + document creations, and (when a message has multiple attachments) N-times-attachment-count document writes; it must stay capped by a fixed per-request max message count (currently 25) and must not be reachable with an unbounded or user-controlled batch size.

### Elevation of Privilege

The highest-likelihood privilege failure here is an authentication bypass that lets an unauthenticated request reach pottery/quilting/travels data — since these apps are fully household-shared by design, there is no per-owner boundary to violate _within_ an authenticated household account, so `requireAuth` (or equivalent session checks) on every route is the control that matters, not per-row ownership filters. The remaining elevation risks are SSRF or injection flaws that let an authenticated user pivot into server-side capabilities beyond the shared-data model (e.g. reaching another user's single-owner Gmail/Calendar tokens, internal network destinations, or raw SQL injection). Raw SQL must remain parameterized, and any server-side fetch or redirect logic must reject internal destinations and untrusted authority changes. Gmail-specific: connection lookups, decisions (dismiss/link/ignore/reconsider), and disconnect must all filter by the session user's own `userId`, never by trip or household membership, since Gmail access itself must remain single-owner even where the resulting trip data is shared. The reconsider endpoint (`POST /gmail/messages/:messageId/reconsider`) reverses a prior ignore/dismiss decision and must only ever touch the requesting user's own decision row (scoped by `userId` + `gmailMessageId`), must reject reconsidering an already-linked message (409), and must 404 when no matching decision exists — it must never be usable to reset or discover another user's decision state. Label management inherits the same constraint: label IDs are cached per-connection row and every `labels.list`/`create`/`modify` call must use the requesting user's own access token, never a token or label ID belonging to another connection.
